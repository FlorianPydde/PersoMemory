ACT II Pillar 1 · Ingestion

Capture everything — judge & reconcile later

① Recall · don't miss anything

Capture broadly

Missing a real task, commitment, or piece of career advice is costly — a bit of noise isn't. Capture everything that might matter; sort signal from noise later.

How · specific prompts

A targeted WorkIQ prompt for each signal — obligations · decisions · risks · career · ideas · mentions. One broad query misses most.

② Reconcile · match work to intent

Asks ≠ work

Asks & decisions live in email, chat & meetings — but the work happens in the terminal (or a doc). No source is complete on its own.

How · session hooks

Each Copilot CLI session drops a pointer to the terminal work; the sweep matches it back to the task it satisfied.

Captured as summaries + source links — never raw transcripts or secrets Sources: M365 · Copilot sessions · "remember this" · repo

ACT II Pillar 3 · Retrieval

When to retrieve — and how to find it

① When · the trigger

Cue-driven, not automatic

Nothing loads at startup. A skill pulls memory only when you ask — your question is the cue.

"What should I focus on?" "What's my career path?" "Have we done this before?"
② How · the search

Follow links

Curated wikilinks between notes form a graph — traverse straight to the related ones. Exact & deterministic; try this first.

Semantic search

Obsidian's Smart Connections finds notes by meaning — great for discovery. Confirm hits with an exact read.

Cue-driven, not session-driven — links first, semantic for discovery. Retrieve the smallest context that helps; more memory makes it worse.
ACT II Pillar 4 · Governance

Where the model can be wrong — and how we catch it

The challenge
How governance handles it

Did this actually get done? The system pairs the work you did (captured by hooks) with the task asked for in chat — but it can pick the wrong pair.

Your tasks live in one file — execution/open-loops.md. The nightly sweep marks the loop done with a link to the proof; if it's unsure, it files a note in governance/approvals/ for you.

Is this worth keeping — and where? The model might exaggerate (call a small win "career evidence") or save it in the wrong place.

It's an approval gate: the proposed entry waits in governance/approvals/ (a dated file) — nothing durable is saved until you approve it in the morning brief.

Is this still current? Projects quietly end and old tasks pile up — nobody tells the system.

A lifecycle service reads each note's frontmatter (status, last-updated, review_date) and flags stale outcomes & aged loops in the morning brief — confirm, close, or extend.

Under the hood the memory skills decide · a lifecycle service scans · it's all plain Markdown in the vault

During the day

How → the router skill writes evidence & gates anything durable

Evening

Sweep

Pull the day's signals, route them, reconcile work vs. tasks, park unclear calls.

How → the sweep skill: multi-prompt recall + reconciliation

Weekly

Consolidate

Step back and distil memory so it stays sharp, not bloated.

How → the maintenance skill

Inside the weekly consolidation
Promote — a pattern reused 3× becomes a durable reusable note.
Close — a project untouched for weeks → done / archived (with your OK).
Prune (decay ≠ delete) — old daily notes are never promoted or linked, so retrieval ignores them; they age out and get archived in the monthly compression.
Merge / supersede — two notes on one topic are combined & their links repointed; replaced guidance sets supersedes: [[old]] and the old note is flagged superseded — kept for history, dropped from recall.
ACT III What actually matters · then the demo

What actually matters

  • Separate raw evidence from interpreted memory.
  • Promote sparingly — most evidence should decay.
  • Retrieve minimally — protect the context window.
  • Govern aggressively — gate high-impact writes.
  • Keep the store human-curated and inspectable.

APPENDIX A Skill internals

Anatomy of a skill & the router

What a "skill" is
  • A SKILL.md instruction module the agent loads on demand — only its name + description stay in context; the body loads when relevant (progressive disclosure).
  • Encodes judgment & policy: when to act, what to read, what to write, what to gate.
  • Lives in ~/.copilot/skills/. Skills decide; MCPs do the I/O.
The router's job
  • Classify intent — recall · capture · attention · maintenance.
  • Two-stage retrieval — deterministic (direct → links → property) then semantic (Smart Connections), confirmed by exact reads.
  • Dispatch — whole-day → brief · intake → sweep · cleanup → maintenance · scoped/ambiguous → handled inline.
  • Enforce approval gates on durable writes.

APPENDIX C Inside the sweep

Recall, the terminal & routing

Recall — 6 targeted WorkIQ passes (candidate lists + source detail, never raw text)
1 · Obligations & requests
promises made, asks to do/review/send/decide.
2 · Outcome changes
status, scope, owner, blocker, risk, priority shifts.
3 · Career & feedback
recognition vs. future guidance, kept separate.
4 · Decisions & risks
decisions, reversals, dependencies, "waiting on".
5 · Reusable artifacts
decks, prompts, patterns, playbooks, lessons.
6 · Direct mentions
@mentions, questions, owner assignments to me.

Plus the terminal

Reads pointer-only entries from the local session-reviews/ queue (dropped by the session-end hook) and extracts loop closures — work in a Copilot session that completed a WorkIQ-raised commitment.

Then routing

Dedupe → write evidence/daily/ → mirror open-loops → update active-now (material only) → queue durable/ambiguous in governance/approvals/. Sources kept separate & attributed.

APPENDIX D Why MCPs?

MCPs: the access layer

The agent's hands, not its brain — each gives a capability the agent has no other way to get. Strip them out and it's blind.

Each earns its place
WorkIQ
The only door into Microsoft 365 — meetings, mail, Teams, transcripts.
mcpvault
Deterministic, frontmatter-aware read/write to the vault — not raw-file fumbling.
smart-connections
Exposes the plugin's embedding index for semantic search.
lifecycle
Deterministic staleness scanner over frontmatter dates.
workiq-teams
Actuation — send Teams messages, on explicit intent only.
Why a tool layer at all
  • Separation of concerns — MCPs are dumb I/O; judgment stays in inspectable skills. Safe to reason about.
  • Standard & swappable — MCP is a protocol; capability is decoupled from the agent.
  • Determinism + a security boundary — reliable writes/checks; each MCP scopes what the agent can touch.

Tools = access · skills = judgment · vault = memory · agent = orchestrator.
